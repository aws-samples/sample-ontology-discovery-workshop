# PRD — Ontology Discovery Workshop Tool

**코드네임:** OntoForge
**작성자:** Solutions Architect / AI Sales Specialist
**상태:** Draft v0.1
**최종 목표:** 고객 디스커버리 워크샵에서 온톨로지를 실시간으로 함께 만들고 검증하여, 후속 기술 미팅(Neptune / Bedrock AgentCore 구현)으로 자연스럽게 연결한다.

---

## 1. 배경 & 문제 정의

세일즈 앞단(Discovery)에서 고객은 자신의 도메인을 **온톨로지(그래프)로 어떻게 표현할 수 있는지** 상상하기 어려워한다. "그래프 DB가 좋다", "에이전트가 추론한다"는 말은 추상적이라 와닿지 않는다.

고객이 확신을 갖고 다음 단계로 넘어가려면 다음을 **눈으로** 봐야 한다.

1. 우리 도메인의 개념(엔티티)과 관계가 그래프로 표현된다 (T-Box, 스키마)
2. 우리 실제 데이터(김민수 학생의 성적, 선생님 피드백 등)가 그 위에 올라간다 (A-Box, 인스턴스)
3. 자연어로 던진 질문이 그래프 쿼리로 풀린다 ("이 답이 진짜 그래프에서 나오네요")
4. 그래서 어떤 데이터가 이미 있고(스키마 매핑), 어떤 데이터를 새로 마련해야 하는지가 정리된다

OntoForge는 이 4가지를 워크샵 현장에서 **대화하면서 실시간으로** 구현해 보여주는 도구다.

### 비목표 (Non-goals)
- 프로덕션 온톨로지 관리 플랫폼이 아니다. (워크샵/PoC 도구)
- 완전한 OWL 추론 엔진을 제공하지 않는다. (§6 제약 참조)
- 최종 운영 그래프 DB가 아니다. 최종 운영은 Neptune으로 승격(promote)한다.

## 2. 사용자 & 시나리오

| 사용자 | 역할 |
|--------|------|
| SA / Sales Specialist (운영자) | 워크샵 진행, 에이전트와 함께 고객 답변을 구조화 |
| 고객 도메인 전문가 | 엔티티·관계·질문을 제공, 그래프를 보며 검증 |

**대표 시나리오 (교육 도메인 예시):**
워크샵 진행 → 에이전트가 "어떤 대상들이 있나요?" 질문 → 학생/학부모/교사/커리큘럼 도출 → 관계 정리(학생-COMPLETED-단원, 교사-FEEDBACK-학생) → 속성·이벤트(점수, 모의고사) 추가 → 고객 질문 리스트 입력 → 갭 분석(빠진 엔티티 자동 도출) → 실제 데이터로 openCypher 검증 → 데이터 소싱 명세 → 기술 미팅으로 인계.

## 3. 핵심 사용자 경험 (워크샵 루프)

매 대화 턴마다 **3개가 동기화**되는 것이 핵심:

```
[대화/받아적기] ──► [구조화 추출(에이전트)] ──► [그래프 DB 반영(Kùzu)]
        │                                              │
        └──────────► [Markdown 문서 출력]              └──► [실시간 그래프 시각화]
```

화면 구성:
- **좌측:** 에이전트 대화 (Claude Code 터미널 또는 채팅 패널). 운영자가 고객 답변을 입력하면 에이전트가 엔티티/관계/속성으로 구조화.
- **우측:** 실시간 그래프 뷰. 엔티티가 추가되거나 관계가 바뀔 때마다 애니메이션으로 갱신. 노드 클릭 시 속성 패널.
- **하단/탭:** 생성되는 Markdown 문서 미리보기 + 실행된 openCypher 쿼리/결과.

## 6. 기술적 제약 (먼저 합의한 사항)

이 도구가 **할 수 있는 것과 못 하는 것**을 명확히 한다. PRD의 가장 중요한 절.

### 6.1 표현 형식: property graph 단일 내부 모델
- 우선순위 합의: ①둘 다 지원(변환 레이어) → ②Property Graph(openCypher) → ③RDF/OWL.
- **현실적 결정:** 내부 단일 진실원천은 **property graph(Kùzu)** 하나로 간다. Neptune이 최종 목표이고 openCypher가 그대로 넘어가기 때문.
- T-Box/A-Box는 *개념*으로 UI에서 분리 표현한다 (스키마 탭 / 인스턴스 탭). 저장은 property graph 메타 레이블·노드 타입으로.

### 6.2 OWL 추론은 범위 밖 (중요한 제약)
- property graph는 OWL의 **공리 기반 추론(reasoner)** — 클래스 계층 추론, 역관계 자동 도출, disjointness 검증 등 — 을 네이티브로 하지 못한다.
- 워크샵에서 "추론"이 필요하면 **명시적 Cypher 쿼리/규칙**으로 흉내 낸다 (예: 전이적 관계는 가변길이 패턴 `-[:PREREQUISITE*]->`).
- 진짜 OWL reasoning이 고객 요건이면, 별도 임베디드 트리플스토어(Oxigraph 등) 병행이 필요하고 이는 도구 범위를 크게 키운다 → **P2(F12)에서 RDF 익스포트만 우선 제공**, 추론은 기술 미팅 의제로 넘긴다.

### 6.3 RDF ↔ Property Graph 손실 변환
- 두 모델은 1:1 매핑이 아니다. property graph의 엣지 속성, RDF의 reification·blank node 등은 변환 시 표현 방식이 달라진다.
- F12 변환 어댑터는 **가장 흔한 패턴(노드→리소스, 엣지→트리플, 속성→리터럴)** 까지만 보장한다. 완전 무손실 왕복(round-trip)은 보장하지 않는다.

### 6.4 Neptune 승격 경계
- 로컬 Kùzu에서 짠 openCypher는 **대부분** Neptune에서 동작하나, 100%는 아니다. (함수·인덱스·일부 절 차이)
- F11 익스포트는 ①openCypher CREATE 스크립트, ②Neptune Bulk Loader용 CSV(노드/엣지) 두 포맷을 낸다. 실제 적재·튜닝은 기술 미팅 단계.

### 6.5 데이터·보안
- 워크샵은 고객 보안망 내 로컬 실행이 기본. 외부로 나가는 것은 Claude API 호출(대화 구조화)뿐 → 고객사 정책상 민감정보가 프롬프트에 들어가면 안 되는 경우, **익명화/마스킹** 또는 온프레미스 모델 옵션이 필요(P1 검토).
- 한국 고객 대상: 생체·개인정보를 다루는 후속 클라우드 구축 시 **PIPA / ISMS-P** 준수가 별도 트랙으로 필요 — 워크샵 도구 자체는 로컬이라 해당 없음, 단 고객 안내 필요.

### 6.6 실행 환경
- 운영자 노트북 1대에서 Python 프로세스 + 로컬 웹서버(브라우저 시각화)로 완결.
- 동시 다중 사용자 협업 편집은 비목표(단일 운영자 모델). 고객은 화면 공유로 본다.

## 7. 워크샵 스킬 정의 (대화 단계)

| 스킬 | 입력 | 출력 |
|------|------|------|
| `extract_entities` | 고객 답변 텍스트 | 엔티티 타입 + 인스턴스 후보(JSON) |
| `define_relations` | 엔티티 목록 + 답변 | 관계(엣지) + 카디널리티 |
| `model_properties` | 엔티티 | 속성/이벤트 노드 정의 |
| `analyze_gaps` | 질문 리스트 + 현재 그래프 | 빠진 엔티티/관계 목록 |
| `verify_query` | 자연어 질문 | openCypher + 실행결과 |
| `map_data_sources` | 현재 그래프 | 보유/미보유 데이터 명세 |

각 스킬은 구조화 JSON을 반환 → 오케스트레이터가 검증 후 Kùzu 반영 + 시각화 브로드캐스트.

## 8. 산출물 (워크샵 종료 시 고객 인계물)

1. **합의된 온톨로지** — Kùzu DB 파일 + openCypher 스키마
2. **검증된 질의 세트** — 자연어 질문 ↔ openCypher ↔ 실제 답 (설득의 핵심 증거)
3. **온톨로지 문서** — Markdown (개념도, 엔티티·관계 사전)
4. **데이터 소싱 명세** — 보유 데이터 스키마 매핑 + 신규 마련 필요 항목
5. **Neptune 익스포트** — openCypher / Bulk Loader CSV
6. **기술 미팅 인계서** — 위 전부 + 오픈 이슈(추론 요건 등)
